Architecture Summary: Hybrid FEC + ARQ Protocol

1. High-Level Design Choices

The core objective of this assignment is to guarantee real-time delivery of audio frames across a hostile UDP network while minimizing playout delay and capping bandwidth overhead at strictly ≤ 2.0x. To achieve this, the architecture utilizes a Hybrid Forward Error Correction (FEC) and Automatic Repeat-Request (ARQ) strategy.

Key Decision: Zero-Jitter Buffer Architecture
Because the testing harness (endpoints.py) judges frame validity purely by whether a packet with the correct sequence number arrived before its absolute deadline (irrespective of network ordering), we explicitly chose to offload jitter absorption entirely to the harness player. Our receiver is completely stateless and bufferless, avoiding complex clock-synchronization and local blocking entirely. Packets are fired straight out to the player the millisecond they are decoded.

2. Forward Error Correction (FEC) Mechanism

Our primary line of defense is proactive redundancy. Every outgoing packet contains the primary frame (164 bytes) and, if the bandwidth budget permits, a redundant payload from a past frame (160 bytes).

  • Offset Selection (k=3): We use a static offset of k=3. This means packet i carries the redundant payload of packet i-3. This offset was deliberately chosen because it mathematically guarantees survival against independent bursts of up to 3 consecutive packet drops, which completely covers the vast majority of real-world packet loss profiles.
  • Delay Tradeoff: To allow k=3 redundancy to be useful, the playout delay (--delay_ms) must be at least 60ms + maximum network jitter. We selected 140ms as our highly competitive submission delay, perfectly clearing 80ms of jitter.

3. Automatic Repeat-Request (ARQ) Fallback

If the hostile network unleashes a massive burst that wipes out a frame AND its redundant backup (bursts > 3 packets), pure FEC fails. We implemented an asynchronous NACK-based ARQ system to catch these edge cases.

  • The receiver maintains a tracking window of arrived sequence numbers. If sequence X+5 arrives but X is still missing, the receiver concludes that the FEC backup is officially lost and fires a 4-byte NACK back over the feedback port.
  • To prevent bandwidth-killing NACK storms, NACKs for the same sequence number are heavily throttled (max once per 80ms).
  • The sender concurrently multiplexes incoming harness frames and incoming NACKs using Go channels, immediately resending requested frames up the network.

4. Dynamic Bandwidth Budget Engine

Maintaining a strict ≤ 2.00x bandwidth overhead is critical. We designed a dynamic, zero-loss budget tracker in the sender.

  • The sender maintains a running sum of the maximum allowed bytes (totalRaw * 2).
  • Whenever an FEC redundant payload OR an ARQ retransmission is sent, it inherently consumes bytes from this budget.
  • Because ARQ retransmissions cost 164 bytes, they organically spike the budget usage. When this happens, the sender is mathematically forced to temporarily withhold appending FEC redundancy to the next few packets until the budget recovers. This creates a beautifully auto-balancing ecosystem where FEC protection is only dialed back when ARQ is actively stealing bandwidth to save a lost frame.

5. Systems-Level Optimizations (Coping with Go GC)

During high-frequency real-time playout, heap allocations that trigger Garbage Collection (GC) pauses are fatal to latency. We heavily optimized the Go implementation to be completely allocation-free in the hot loops:

  • Ring Buffers over Maps: We eradicated a massive memory leak by replacing the receiver's tracking map[uint32]time.Time with a statically allocated fixed-size array ([256]time.Time), granting bare O(1) access and zero allocations.
  • Value-Type Channels: Instead of allocating dynamic byte slices (make([]byte, 160)) 50 times a second for the sender's harness channel, we implemented a custom struct with a fixed array (Payload [160]byte) passed directly by value.
  • CPU Throttling: The NACK evaluator logic was decoupled from the per-packet arrival loop and restricted via a time.Now() delta check, restricting CPU-heavy array iterations to a maximum of 100Hz.
